# ПЛАН Viral Monitor — разбивка на модули и последовательность разработки

## Контекст

Документ описывает полную функциональную и техническую разбивку платформы Viral Monitor на самостоятельные юниты разработки. Разбивка максимально дробная — каждый модуль это отдельная бизнес- или техническая ответственность, которую можно реализовать и протестировать изолированно. Это **не** означает «один модуль = один контейнер»: группировка в сервисы и контейнеры — отдельный раздел (D).

Исходные требования: [ТЗ_ВИРАЛ-монитор.md](ТЗ_ВИРАЛ-монитор.md).

Текущее состояние:
- ✅ Реализован [Modules/processor](Modules/processor/) — FastAPI + SQLite + внешние AI API (AssemblyAI/Deepgram/OpenAI Whisper/Groq для транскрипции; Claude/GPT-4o/Gemini для vision). Работает изолированно, принимает локальные файлы.
- ⬜ [Modules/downloader](Modules/downloader/) — пустая заготовка.
- ⬜ Остальное — только в ТЗ.

Легенда: ✅ реализовано, 🟡 частично, ⬜ не реализовано.

## D. Предварительная группировка в docker-контейнеры

Принципы:
- Разные типы ресурсов — разные контейнеры (CPU ≠ GPU ≠ IO).
- Разный масштабируемый профиль — разные контейнеры.
- Одна доменная граница — один контейнер, даже если внутри несколько юнитов разработки.
- Общая БД и общий стек → можно клеить.
- `processor` остаётся изолированным — не расширяем его скриптами/паблишингом.

Целевая композиция (стартовая гипотеза, обсуждается):

| Контейнер | Входящие модули | Ресурс |
|---|---|---|
| **processor** ✅ | A3 (video analysis) + A4 (LLM layer, частично) | CPU |
| **downloader** | A1 целиком | IO |
| **crawler** | A2 (Scheduler + Metrics + Trending + Dedup) | CPU |

Итого: **~9 прикладных + ~7 инфраструктурных контейнеров**. Финальное решение — после согласования каталога A/B.
